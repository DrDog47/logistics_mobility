## Mobility Payroll · версия 1.0 (черновик для правки · 2026-06-08)

*Логистическая компания (г. Белосток, Польша)*

> Документ описывает **продуктовые требования** к модулю учёта водителей: какие данные хранятся, какие действия доступны, кому и по каким правилам. Техническая схема хранения — в [`TDD_data_schema_v2.0.md`](TDD_data_schema_v2.0.md). Общий контекст системы учёта документов — в [`TZ_sistema_ucheta_dokumentov_v1.4.md`](TZ_sistema_ucheta_dokumentov_v1.4.md).

## 1. Контекст и цель

Компания занимается грузоперевозками (автопарк ~24 фуры). Водители нанимаются преимущественно из Беларуси; для легальной работы в ЕС у каждого должен быть действующий легализационный документ с контролируемым сроком. Модуль «Водители» — это единый реестр водителей: карточка с персональными данными, привязка к организации-работодателю, история контрактов и пакет документов с контролем сроков.

### 1.1. Целевой результат

- Единая карточка водителя со всеми идентификационными данными и контактами.
- Привязка водителя к организации (§3 PRD-схемы) и к истории трудовых контрактов.
- Пакет документов водителя с автоматическим расчётом сроков и подсветкой истекающих (см. модуль документов).
- Минимум ручного ввода при загрузке документов: пакетная загрузка перетаскиванием (drag-and-drop) — см. §8.

## 3. Карточка водителя

Сущность **Водитель** не зависит от конкретного контракта (контракты — отдельная история, §7).

### 3.1. Поля

| Поле | Обяз. | Правило / формат | Описание |
|------|:----:|------------------|----------|
| Имя (`first_name`) | да | до 64 симв. | Латинское написание как в паспорте |
| Фамилия (`last_name`) | да | до 64 симв. | Латинское написание как в паспорте |
| Дата рождения (`birth_date`) | да | дата | — |
| Гражданство (`nationality`) | да | ISO 3166-1 alpha-3, по умолч. `POL` | Выбор из справочника стран |
| Национальный ID (`identification_id`) | да | до 30 симв., **уникален** | Бизнес-ключ водителя (как правило — номер паспорта) |
| PESEL (`pesel`) | нет | ровно 11 цифр, **уникален** | Только для резидентов Польши |
| Номер паспорта (`passport_number`) | нет | до 32 симв. | — |
| Номер карты тахографа (`tachograph_card_number`) | нет | до 20 симв., **уникален** | — |
| Телефон (`phone`) | нет | до 32 симв. | Контакт |
| Заметки (`notes`) | нет | до 1000 симв. | Свободный текст |
| Дата найма (`hire_date`) | да | дата, по умолч. — сегодня | — |
| Дата увольнения (`termination_date`) | нет | дата | — |
| Активен (`is_active`) | — | по умолч. `да` | Признак активности водителя |
| Организация (`organisation_uuid`) | да | выбор из списка организаций | Работодатель |
| `extra` | нет | JSON | Резерв под расширяемые атрибуты (PRD §3.4) |

> **Полное имя** отображается как `Имя Фамилия`; отдельного поля нет (вычисляется).

### 3.2. Связи

- **Организация** — каждый водитель относится к одной организации; организацию нельзя удалить, пока на неё ссылаются водители (FK `RESTRICT`).
- **Контракты** — ноль или несколько (§7), сортируются по дате начала (новые сверху).
- **Документы** — ноль или несколько (паспорт, виза/ВНЖ, и т.д.), сортируются по дате окончания.

## 4. Сценарии использования

### 4.1. Список водителей
Любой аутентифицированный сотрудник видит список, отсортированный по фамилии, имени.

### 4.2. Живой поиск
Поле поиска фильтрует по подстроке (без учёта регистра) по полям: фамилия, имя, PESEL, номер паспорта, национальный ID. Поиск работает «на лету» (без перезагрузки страницы — HTMX-фрагмент таблицы).

### 4.3. Создание водителя
Форма со всеми полями §3.1. После сохранения — переход на карточку водителя, сообщение «Водитель … добавлен».

### 4.4. Просмотр карточки
Карточка показывает: идентификацию, организацию, сводку обязательных документов (с бейджами сроков), пакет документов и панель отпусков. Детальная структура — в §4.7.2. *Историю контрактов отдельной секцией заменила сводка обязательных документов (§12.5); контракт водителя — это документ-трудоустройство в общем пакете документов, см. §7.*

### 4.5. Редактирование
Изменение полей карточки. Сообщение «Водитель обновлён».

### 4.6. Удаление (мягкое)
Удаление помечает запись как удалённую (soft delete), физически не стирает. Удалённые водители не показываются в списке/поиске и недоступны по прямой ссылке (404).

### 4.7. Текущая структура страниц (как реализовано на 2026-06-10)

> Раздел фиксирует **фактическую** вёрстку двух страниц модуля — что и в каком порядке выводится. Источник: `app/drivers/routes.py`, шаблоны `app/templates/drivers/`. Маршруты обслуживает blueprint `drivers` (см. §6).

#### 4.7.1. Страница списка — `GET /drivers/` (`list_drivers` → `drivers/list.html`)

Доступна любому аутентифицированному сотруднику. Сверху вниз:

1. **Заголовок страницы** — `H1 «Drivers»` и кнопка **«+ New driver»** (только `admin` / `fleet_manager`; ведёт на §4.3).
2. **Полоса KPI** (`.kpis`, счётчики по всему автопарку, не по текущему фильтру поиска — функция `_driver_stats()`):
   - **Total drivers** — всего активных водителей.
   - **Active contracts** — водители с действующим документом-трудоустройством (`employment`, по `start_date ≤ сегодня` и `end_date` пусто/в будущем); подсказкой — «N without contract».
   - **Docs expiring** — число документов в зоне внимания (просрочен / `critical` / `urgent` / `soon`); если > 0 — это раскрывающийся список (`<details>`) с переходами на карточки и бейджем уровня; если 0 — простой счётчик.
   - **On leave now** — число водителей в отпуске сегодня; если > 0 — раскрывающийся список «кто и до какой даты», иначе счётчик 0.
3. **Панель со списком** (`.panel`):
   - **Тулбар поиска** — поле `search`, живой поиск через **HTMX** (`hx-get` на `/drivers/` с `hx-trigger="input changed delay:300ms"` и по Enter), цель — фрагмент `#drivers-table`, замена `outerHTML`. Поиск по подстроке: фамилия, имя, PESEL, номер паспорта, национальный ID (§4.2).
   - **Таблица водителей** (партиал `drivers/_table.html`) со столбцами: **Name** (ссылка на карточку) · **Nationality** (pill) · **Organisation** (ссылка на организацию) · **Contract** (pill типа контракта или «—») · **Documents** (число активных) · **Expiring documents** (бейдж числа в зоне внимания или 0) · **Hired** (дата) · **действия** (View, и для `admin`/`fleet_manager` — Delete с подтверждением). Пустой список → строка «No drivers found». Счётчики документов считаются одним запросом на всех показанных водителей (`_document_counts()`, без N+1).
4. **Удалённые водители** (только при полном рендере, не в HTMX-фрагменте) — если есть soft-deleted записи, отдельный заголовок «Deleted drivers (N)» и таблица-архив (партиал `drivers/_deleted_table.html`) с возможностью восстановления (`restore_driver`).

> **HTMX-контракт:** при заголовке `HX-Request` маршрут возвращает **только** `drivers/_table.html` (живой поиск), иначе — полную страницу со статистикой и архивом.

#### 4.7.2. Карточка водителя — `GET /drivers/{uuid}` (`show_driver` → `drivers/show.html`)

Доступна любому аутентифицированному сотруднику; активный водитель или `404`. Контекст включает данные панели отпусков (`build_panel_context`) и формы добавления отпуска/лимита. Структура:

1. **Хлебные крошки** — кнопка «← Back to drivers» и путь «Drivers › Employee card».
2. **Верхний блок** (`.panel.top-card`) — две колонки, разделённые вертикальным делителем:
   - **Левая — профиль** (`.profile-section`):
     - Шапка: аватар (инициалы), имя, тег национальности (если задан), тег «inactive» (если водитель неактивен).
     - **Passport** — с бейджем `EU` / `non-EU` по типу активного паспорта; поля: Passport №, National ID, Nationality, Birth date.
     - **Legal entity** — Organisation (ссылка), Hire date.
     - **Polish identifiers** — PESEL, Tachograph card.
     - **Contact** — Phone.
     - Кнопка **«✎ Edit profile»** (только `admin` / `fleet_manager`, §4.5).
   - **Правая — сводка обязательных документов** (`reqs-section`, чистая функция `driver_completeness`, заменяет прежнюю секцию истории контрактов — §12.5): общий статус (**Complete** / **N missing** / **N expiring**) и блоки **Missing** (теги типов), **Expiring soon** (строки с бейджем дней и датой, ссылка на редактирование документа) и **In order** (теги).
3. **Нижний блок** (`.driver-bottom`, во всю ширину):
   - **Документы** (`.panel`): заголовок с действиями **«Download all (.zip)»** (если есть документы) и **«+ Add document»** (роли-редакторы); таблица текущих документов (партиал `documents/_doc_table.html`, без документов-контрактов); зона **drag-and-drop** для загрузки (партиал `documents/_dropzone.html`, роли-редакторы, §8); при наличии — секция **Archived documents** со своей таблицей и zip-выгрузкой.
   - **Vacations & Schedule** (`.panel`): панель отпусков (партиал `vacations/_panel.html`) — календарь месяца с навигацией через `?vac_month`, баланс дней, список отпусков, формы добавления отпуска и редактирования лимита. *У ТС этой секции нет (PRD ТС §1.2).*

> **Связанные маршруты** (не на самой странице, но из её действий): `edit_driver`, `delete_driver`, `restore_driver`; контракты как документы-трудоустройства — `add_contract` / `edit_contract` / `delete_contract` (§7).

## 5. Бизнес-правила

1. **Уникальность.** `identification_id`, `pesel`, `tachograph_card_number` — уникальны в пределах системы. Повторная запись с тем же значением не создаётся.
2. **Обязательная организация.** Водитель не может существовать без организации-работодателя.
3. **Мягкое удаление.** Записи не стираются; каждый список/поиск/карточка работают только с активными (`is_deleted = false`). Удаление обратимо на уровне данных.
4. **PESEL — только для резидентов Польши**, поле опционально и проверяется по маске «11 цифр».
5. **Даты в UTC** на уровне хранения; в интерфейсе — локальная дата.

## 7. Контракты водителя

Водитель может иметь несколько контрактов во времени. Контракт — отдельная сущность.

### 7.1. Поля контракта

| Поле | Обяз. | Правило | Описание |
|------|:----:|---------|----------|
| Тип (`contract_type`) | да | `umowa_o_prace` / `umowa_zlecenia` / `b2b` | Тип договора по польскому праву |
| Дата начала (`start_date`) | да | дата | — |
| Дата окончания (`end_date`) | нет | дата | Пусто = открытый (бессрочный) |
| Базовая ставка PLN (`base_salary_pln`) | да | ≥ 0, 2 знака | Брутто для умовы; согласованная ставка для B2B |
| Норма часов в месяц (`hours_norm`) | да | 0–300, по умолч. 168 | Для договоров по КП; для B2B игнорируется |

### 7.2. Правила

- **Активный контракт** — тот, у которого `start_date ≤ сегодня ≤ end_date` (или `end_date` пусто). Используется расчётом зарплаты (модуль payroll, сейчас «припаркован»).
- Контракты отображаются от новых к старым.

## 8. Планируемое в v1.0 — массовая загрузка документов (drag-and-drop)

> Требование №1. Связано с подходом из [`TZ_sistema_ucheta_dokumentov_v1.4.md`](TZ_sistema_ucheta_dokumentov_v1.4.md) (распознавание типа документа и сроков), перенесённым внутрь веб-приложения.

### 8.1. Что нужно

Добавить **зону drag-and-drop** в **двух местах** (решение по §9 п.4):

- в **общем разделе «Водители»** — для смешанного пакета документов разных водителей (водитель определяется распознаванием по каждому файлу);
- на **карточке конкретного водителя** — этот водитель предлагается по умолчанию для всех файлов, с возможностью переопределить привязку отдельных файлов.

В обе зоны можно перетащить **большой пакет документов сразу** (множественная загрузка файлов).

> **Статус реализации.** UI-зона drag-and-drop **реализована** в обоих местах: `app/templates/documents/_dropzone.html` (партиал), подключена в `drivers/list.html` (общий раздел) и `drivers/show.html` (карточка, водитель по умолчанию); клиентская логика — `app/static/js/dropzone.js`, стили — в `app/static/css/style.css`. **Endpoint загрузки реализован** (`POST /documents/upload`, blueprint `documents`): принятый пакет сохраняется в `_Inbox/` (фаза 1, §8.6), без записи в БД и без распознавания. Путь хранилища настраивается в конфиге (`DOCUMENTS_DIR`, имя inbox — `DOCUMENTS_INBOX_DIRNAME`, разрешённые расширения — `DOCUMENTS_ALLOWED_EXTENSIONS`). Распознавание/привязка/запись (§8.4–8.5) и экран подтверждения — следующий этап.

> **Требование №2.** Пакет может быть **смешанным** — содержать документы **разных водителей** одновременно. Поэтому привязка определяется **для каждого файла отдельно**, а не одним общим выбором на весь пакет.

### 8.2. Пользовательский сценарий

> **Статус: реализовано.** Распознавание (`POST /documents/inbox/recognize`) рендерит **редактируемую форму** (`documents/_inbox_edit.html`): по каждому файлу можно поправить тип, ФИО, `identification_id`/паспорт, даты, а также **вручную привязать файл к водителю** (override авто-сопоставления) или отметить «skip». Сохранение (`POST /documents/inbox/apply`) применяет именно отредактированные данные (без повторного вызова модели). Парсер формы — `routes._entries_from_form`; ручная привязка — параметр `forced` в `apply_recognized`.

1. Пользователь открывает зону загрузки и перетаскивает в неё пакет файлов (сканы паспортов, виз, ВНЖ, техосмотров и т.п.) — файлы могут относиться к **разным водителям**.
2. Файлы **сначала загружаются в папку `_Inbox/`** (смонтированный том, §8.6) и только **затем обрабатываются** — приём пакета отделён от распознавания. Система показывает список загруженных файлов со статусом.
3. Для каждого файла система пытается **распознать**: тип документа, ключевые даты (срок действия / окончания) и **к какому водителю он относится** (по имени/идентификатору в распознанных данных).
4. Система **группирует** файлы по предполагаемым водителям и показывает результат на одном экране подтверждения: «файл → водитель → тип → даты».
5. Пользователь **подтверждает или правит** по каждому файлу: привязку к водителю, тип, даты. Нераспознанные/неоднозначные файлы помечаются и требуют ручного выбора водителя.
6. После подтверждения для каждого файла создаётся запись документа соответствующего водителя (`DriverDocument`) с привязкой к ссылке на файл; сроки сразу попадают в контроль (бейджи 120/60 дней, для страховки — 60/30/15).

### 8.3. Требования к функции

- **Множественная загрузка**: один жест перетаскивания принимает много файлов; для «большого пакета» — без жёсткого ограничения на единичную операцию (конкретный лимит уточняется).
- **Смешанный пакет (несколько водителей)**: привязка определяется **по каждому файлу**. Один пакет может создать документы для нескольких водителей за один проход.
- **Привязка к водителю**:
  - при загрузке **с карточки водителя** — этот водитель предлагается по умолчанию, но привязку можно переопределить для отдельных файлов (на случай, если в пакет попали чужие документы);
  - в **общем разделе** — водитель для каждого файла определяется распознаванием, с возможностью ручной правки.
- **Сопоставление с существующими водителями**: распознанного водителя система сопоставляет с реестром по бизнес-ключам (национальный ID / паспорт / ФИО). Если совпадения нет — файл выносится в группу «новый/неопознанный водитель» (поведение для таких случаев — открытый вопрос, §9).
- **Подтверждение перед записью**: ни одна запись в реестр не создаётся без шага подтверждения пользователем.
- **Права**: загрузка документов доступна ролям admin / accountant / fleet_manager (как и обычное добавление документа, §6).
- **Поддерживаемые форматы**, способ хранения файлов (внешняя ссылка vs загрузка в хранилище) и движок распознавания — **открытые вопросы**, см. §9.

### 8.4. Логика по распознанному паспорту: создать / обновить / пропустить

> **Требование №3.** Распознанный документ-**паспорт** управляет не только записью документа, но и **самой карточкой водителя**.

**Ключ сопоставления — стабильный идентификационный (личный) номер** водителя, поле `identification_id`. В большинстве типов паспортов он так и подписан («identification number» / личный номер) и **сохраняется при перевыпуске** книжки (по возрасту, при утере). Номер паспортной книжки (`passport_number`) для сопоставления **не используется как надёжный ключ** — он меняется и распознаётся менее стабильно.

> Стабильный ключ — `identification_id` (уникальное поле). Номер/серия и срок паспортной книжки — отдельные поля (`passport_number` и т.п.) и при перевыпуске обновляются, не затрагивая `identification_id`.

> **Примечание для разработки.** Комментарий в коде модели (`app/drivers/models.py`: «identification_id = passport number») вводит в заблуждение — его следует поправить: `identification_id` — это **личный идентификационный номер**, а не номер паспортной книжки.

По результату распознавания паспорта система выбирает одно из трёх действий:

| Условие | Действие |
|---------|----------|
| Водителя с таким идентификационным номером **нет** в базе | **Создать нового водителя** из распознанных данных паспорта (ФИО, дата рождения, гражданство, идентификационный номер, номер и срок паспорта) |
| Водитель **есть**, и данные паспорта **изменились** (новый номер/серия или срок — перевыпуск по возрасту, после утери) | **Обновить** данные паспорта у существующего водителя (номер книжки, срок), идентификационный номер остаётся прежним |
| Водитель **есть**, и данные паспорта **совпадают** с уже сохранёнными | **Ничего не делать** (no-op) — повторная загрузка того же паспорта не плодит дублей и не меняет карточку |

Требования к этой логике:

- Действие (создать / обновить / пропустить) показывается пользователю **на экране подтверждения** до записи; пользователь может его отменить или поправить привязку.
- При **создании нового водителя** заполняются только данные, распознанные из паспорта (ФИО, дата рождения, гражданство, идентификационный номер, номер и срок паспорта).
- **Организация (`organisation_uuid`) при автосоздании НЕ присваивается** и остаётся пустой: из пакета документов компания, как правило, не определяется — её заполняют **вручную** позже. То есть для автосозданного водителя привязка к организации необязательна на этапе загрузки.

> **Правка модели (реализовано).** `organisation_uuid`, `hire_date` (миграция `b2c3d4e5f6a7`), а также `birth_date` и `nationality` (миграция `c3d4e5f6a7b8`) у водителя теперь **nullable** (`app/drivers/models.py`) — автосозданный из паспорта водитель существует с тем, что распозналось, остальное заполняется вручную. *Осталось (UI):* фильтр/пометка «без организации» в списке водителей, чтобы такие записи было видно и легко дозаполнить.
- При **обновлении** прежний паспорт сохраняется в истории документов (новый скан добавляется как новый `DriverDocument`, старый не теряется); меняются только поля карточки, относящиеся к паспорту.
- Логика применяется и в смешанном пакете (§8.2) — независимо по каждому распознанному паспорту.

### 8.5. Привязка остальных документов и определение типа

> **Требование №4.** После обработки паспортов (§8.4) система проходит по **остальным** документам пакета.

Порядок обработки пакета:

1. **Сначала паспорта** — создают/обновляют карточки водителей (§8.4), так что к этому шагу нужные водители уже есть в базе.
2. **Затем остальные документы** — каждый привязывается к существующему водителю и классифицируется по типу.

Для каждого не-паспортного документа система:

- **извлекает идентифицирующие данные**: имя, фамилию и, если присутствуют, паспортные данные — идентификационный номер (`identification_id`) и/или номер паспорта (`passport_id` / `passport_number`);
- **сопоставляет с водителем** в реестре по приоритету:
  1. по `identification_id` (если распознан) — основной и надёжный ключ;
  2. иначе по совпадению **имя + фамилия** (при неоднозначности файл помечается «требует ручного выбора»);
  3. `passport_number` используется только как **вспомогательная подсказка** для подтверждения совпадения, но **не как самостоятельный ключ** (ненадёжен, §8.4);
- **определяет тип документа** (виза, ВНЖ/legalization, техосмотр и т.п.) — см. §8.5.1;
- **определяет даты действия** документа (начало и окончание) — см. §8.5.2;
- если есть, извлекает номер самого документа (`document_id`);
- создаёт запись `DriverDocument` у найденного водителя — **после подтверждения** на общем экране (§8.2).

#### 8.5.0. Модель «документ ↔ файлы»

Один документ (`DriverDocument`) может состоять из **нескольких файлов** (лицевая/оборотная стороны, страницы) — drag-n-drop добавляет в документ файлы, соответствующие именно этому документу. Файлы хранятся строками в отдельной таблице **`driver_file`** (один-ко-многим, FK `document_uuid → driver_document`; см. TDD §8a); у каждого файла своя распознанная мета (тип/номер/даты) и ссылка на хранилище `file_link`.

- **Документ создаётся вместе с первым файлом.** Если тип файла распознан, а документа ещё нет, при записи файла сначала создаётся `DriverDocument`, затем его `uuid` подставляется в `driver_file.document_uuid` — в рамках одной транзакции Postgres.
- **Файл без документа — на ручную обработку.** Если файлу не удаётся присвоить `document_uuid` (не определены водитель или документ автоматически), файл **остаётся в `_Inbox/`** для ручного разбора, строка `driver_file` не создаётся.

#### 8.5.1. Сопоставление типа документа с каталогом (требование 4.1)

Распознанный тип документа нужно **сопоставить с уже существующими типами** из каталога `document_type` (таблица с парой-ключом `(type, entity_type)`, для водителя `entity_type = "driver"`):

- система нормализует распознанный тип и ищет **соответствующую запись** в каталоге среди активных строк (`is_deleted = false`);
- при **нахождении соответствия** документу присваивается код этого типа (`DriverDocument.document_type` = `document_type.type`);
- сопоставление выполняется по коду типа (`passport`, `visa`, `residence`, `license`, `code95`, `medical`, `tacho_card`, `adr`, `employment`, … — базовый список в `app/documents/constants.py`), а распознанный текст приводится к нему через синонимы/ярлыки.

> **Жёсткое ограничение модели.** `driver_document` ссылается на `document_type` составным внешним ключом с `ondelete RESTRICT`. Поэтому документ **невозможно записать с типом, которого нет в каталоге** — присвоить можно только уже существующий тип. Это и есть суть требования 4.1.

**Если соответствие не найдено** (тип не распознан или его нет в каталоге) — документ **не записывается** и **остаётся в `_Inbox/`** для ручного разбора (решение по §9 п.9), аналогично нераспознанным файлам (§8.6). Автосоздание новых типов в каталоге в v1.0 не выполняется.

#### 8.5.2. Определение дат действия документа (требование 4.2)

Из документа извлекаются **дата начала** и **дата окончания** действия и записываются в поля `DriverDocument`:

- `start_date` — дата начала действия (выдачи / «valid from»);
- `end_date` — дата окончания действия (срок / «valid to»).

Правила:

- **Оба поля опциональны** в модели. У части документов есть только дата окончания (например, паспорт, техосмотр) — тогда `start_date` остаётся пустым; у документов-периодов (виза, ВНЖ, страховка) распознаются обе даты.
- **`end_date` — ключ контроля сроков.** Именно по нему считаются бейджи приближения срока: общая шкала 120/60 дней, для страховки ТС — 60/30/15. Типы из списка «не отслеживаемых» (`UNTRACKED_TYPES`: `employment`, `pesel`, `tech_passport` и т.п.) не подсвечиваются, даже если дата есть.
- Если у документа типа с контролем срока **дата окончания не распознана** — документ помечается «нет даты окончания» и выносится на ручную проверку при подтверждении (срок без даты контролировать нельзя).
- При **проверке корректности**: если распознаны обе даты и `end_date < start_date`, пара считается ошибочной и требует ручной правки.
- Даты показываются в превью и редактируются пользователем до записи.

#### 8.5.3. Запись в таблицу `driver_document` (требование 4.3)

После подтверждения для каждого документа создаётся строка в таблице `driver_document`, **привязанная к `uuid` нужного водителя** (того, что найден §8.5 или создан/обновлён паспортом §8.4).

Состав записи:

| Поле | Значение |
|------|----------|
| `uuid` | Генерируется автоматически (PK записи документа) |
| `driver_uuid` | **UUID водителя**, которому соответствует документ (FK → `drivers.uuid`) |
| `entity_type` | `"driver"` (константа; вместе с `document_type` образует составной FK в каталог) |
| `document_type` | Код типа из каталога (§8.5.1) |
| `document_id` | Номер самого документа, если распознан (иначе пусто) |
| `start_date` / `end_date` | Даты действия (§8.5.2) |
| `file_links` | Ссылка(и) на скан(ы) — JSON-массив |
| `extra` | Тип-специфичные атрибуты, если есть (иначе пусто) |

Правила записи:

- **Транзакционность пакета.** Запись подтверждённых документов выполняется единым проходом; при создании/обновлении водителей паспортами (§8.4) к моменту вставки документов их `uuid` уже известны, и каждый документ ссылается на правильного водителя — в том числе в смешанном пакете (документы разных водителей в одной загрузке).
- **Целостность ссылок.** `driver_uuid` обязан указывать на существующего активного водителя; `document_type` обязан существовать в каталоге (составной FK `RESTRICT`, §8.5.1) — иначе вставка отклоняется и документ остаётся в «требует внимания».
- **Контроль сроков** включается сразу после записи: бейджи рассчитываются по `end_date` (§8.5.2) и появляются на карточке водителя.

**Защита от дублей (решение по §9 п.10).** Перед записью система ищет у водителя **существующий документ того же типа** (`document_type`) **с теми же сроками действия** (`start_date` / `end_date`):

- **Дубль найден, данные водителя не изменились** → **ничего не менять** (no-op): повторная загрузка того же документа не создаёт вторую запись и не правит карточку.
- **Дубль найден, но данные водителя изменились** (в распознанном документе обновились сведения о водителе) → **обновить** данные.
- **Дубля нет** (нет документа этого типа с такими сроками) → создать новую запись (§8.5.3).

> Под «данными водителя» здесь понимаются сведения, извлечённые из документа и относящиеся к карточке/документу (ФИО, `identification_id`, паспортные данные и т.п.). Добавление **второй стороны/страницы** того же документа — это не дубль, а пополнение `file_links` существующей записи (§8.5.4).

#### 8.5.4. Несколько файлов на один документ

> **Статус: реализовано.** Файлы одного пакета с одинаковыми «водитель + тип + сроки» (а при отсутствии дат — с одинаковым нормализованным именем без токенов front/back/page) группируются в одну запись `driver_document` с несколькими путями в `file_links` (`app/documents/persistence.py:_group_units`). Дозагрузка скана к уже существующему документу — тоже есть.

Один **логический документ** может состоять из **нескольких загруженных файлов** — например, лицевая и оборотная стороны, или многостраничный скан. Такой документ — это **одна запись** `driver_document`, а её сканы хранятся **списком** в поле `file_links` (JSON-массив ссылок).

Правила:

- При загрузке несколько файлов, относящихся к одному документу, **группируются в одну запись**, а не создают дубли. **Правило v1.0 (решение по §9 п.11):**
  - **автогруппировка** по «**водитель + тип + сроки**» (совпали → один документ, файлы складываются в `file_links`);
  - дополнительно — по **названию/номеру самого документа** (`document_id` / имя файла), если он распознан и совпадает;
  - плюс **ручное** объединение/разъединение на экране подтверждения;
  - распознавание парности «front/back» — **позже** (вне v1.0).
- На **экране подтверждения** (§8.2) пользователь видит сгруппированные файлы под одним документом и может **объединять/разъединять** их вручную (перенести лишний файл в отдельный документ или, наоборот, добавить сторону к существующему).
- Порядок файлов в `file_links` сохраняется (например, сначала лицевая сторона, затем оборотная).
- Дозагрузка стороны к **уже существующему** документу водителя добавляет ссылку в его `file_links` (а не создаёт новый документ) — связано с защитой от дублей, §9 п.10.

Требования к этому шагу:

- В v1.0 не-паспортный документ **не создаёт** нового водителя (триггер создания — только паспорт, §8.4). Если водитель не найден ни одним способом — файл **остаётся в `_Inbox/`** для ручного разбора, в реестр не вносится (§8.6).
- Тип документа и привязка показываются в превью и редактируются пользователем до записи.
- Документ привязывается к тому же водителю, что был создан/обновлён паспортом из этого же пакета.

### 8.6. Хранение файлов (локально, docker volume) — решение по §9 п.1

> **Статус: реализовано.** Endpoint `POST /documents/upload` (blueprint `documents`, роли admin/fleet_manager/accountant) сохраняет принятый пакет в `<DOCUMENTS_DIR>/_Inbox/` без записи в БД и без распознавания. Путь и параметры — в конфиге: `DOCUMENTS_DIR` (для локального запуска — в `.env`; в docker — env + том), `DOCUMENTS_INBOX_DIRNAME`, `DOCUMENTS_ALLOWED_EXTENSIONS`. Сервис — `app/documents/services.py:save_uploads_to_inbox()`.

В v1.0 файлы сканов хранятся **локально на файловой системе**, а не в облаке/объектном хранилище.

- **Монтирование папки в контейнер.** Внешняя папка с документами подключается к docker-образу через **volume mount**, чтобы при старте приложение **ссылалось на уже существующие файлы**, организованные по структуре из ТЗ. Папка — **на запись** (приложение сохраняет новые файлы и переносит устаревшие в `Archive/`), в отличие от текущего `./data:/app/data:ro` (read-only).
- **Двухфазная загрузка через `_Inbox/` (решение по §9 п.7).** Папка `_Inbox/` (тоже смонтированная — часть того же тома документов) — это **точка приёма**: файлы из drag-and-drop **сначала попадают в `_Inbox/`** и только **позже обрабатываются** (распознавание, привязка, раскладка). Это разделяет «загрузку» и «обработку»: приём пакета не блокируется распознаванием.
- **Что остаётся в `_Inbox/`.** Файлы, которые **не удалось обработать автоматически** (не распознан тип; не опознан водитель у не-паспортного документа; «не наш документ»), **остаются в `_Inbox/`** для **ручного разбора** — не раскладываются в карточки и не вносятся в реестр, пока человек не обработает их вручную.
- **Конфигурация.** Путь к корню хранилища задаётся переменной окружения (например, `DOCUMENTS_DIR`) и пробрасывается в `docker-compose.yml` как volume; путь внутри контейнера читается приложением при старте.
- **Структура и именование — по ТЗ** ([`TZ_sistema_ucheta_dokumentov_v1.4.md`](TZ_sistema_ucheta_dokumentov_v1.4.md), §3). Новые файлы **сохраняются в эту же иерархию**:

```
<DOCUMENTS_DIR>/
├── _Inbox/                       ← точка входа для новых сканов
├── Drivers/
│   └── <Name_Surname>/           ← латиницей из паспорта
│       ├── <Name_Surname>_<Type>_<dates>.<ext>
│       └── Archive/              ← устаревшие версии (не удаляются)
└── Transport/
    └── <Vehicle>/ …
```

- **Правило именования файлов (из ТЗ §3).** `<Владелец>_<Тип>_<дата(ы)>.<ext>`; даты в формате `ГГГГ-ММ-ДД`; документы со сроком получают две даты `начало_конец`. Имена файлов и папок — латиницей/по-английски. Примеры: `Ivan_Ivanov_Passport_2030-05-12.pdf`, `Ivan_Ivanov_Legalization_visa_2024-06-01_2026-06-01.pdf`.
- **Связь с БД.** В `driver_document.file_links` хранятся **пути/ссылки на файлы внутри этой иерархии** (относительно `DOCUMENTS_DIR`), а не загруженные blob'ы. Несколько сторон одного документа — несколько путей в массиве (§8.5.4).
- **Версионность / сосуществование документов (решение по §9 п.12, реализовано).** Реализация — `app/documents/persistence.py` (`_run_archive_pass`, `_archive_document`) + поле `driver_document.archived_at` (миграция `d4e5f6a7b8c9`); архивные документы исключаются из `Driver.active_documents`. Если у субъекта есть **два документа, отвечающие за одно и то же** (один тип/назначение, например две действующие легализации или два полиса), **оба сохраняются действующими** до момента истечения **раньше истекающего** из них:
  - пока оба действуют — **оба хранятся** в карточке/папке водителя и доступны; в `Archive/` не уходят;
  - когда раньше истекающий документ **истёк** — он переносится в `Archive/`, а второй (более поздний по `end_date`) остаётся действующим;
  - **основным в реестре** (для контроля сроков и алертов) считается документ с **самой поздней датой окончания** (`end_date`).
  - Уже **просроченный** при загрузке документ, если есть более свежая действующая версия, кладётся сразу в `Archive/` и в реестр не вносится (из ТЗ).

> **Замечание по инфраструктуре.** Локальное хранилище привязывает приложение к конкретному хосту/тому: при переносе на другой сервер том с документами нужно переносить вместе. Резервное копирование тома — вне рамок PRD, но должно быть учтено в эксплуатации.

## 12. Контроль комплектности и истечения сроков (подход)

> **Открытый вопрос пользователя.** Реализовывать контроль (наличие полного набора §11 + сроки) через существующую БД или иным подходом, учитывая, что **правила меняются и расширяются** со временем?

**Рекомендация: разделить три слоя и хранить *правила* как версионируемую регуляторную конфигурацию, а не в коде и не в админ-таблицах БД.**

### 12.1. Три слоя

1. **Факты — остаются в БД (без изменений).** Конкретные документы водителя с датами — это таблица `driver_document` (§8.5.3). Это фактический реестр, ему место в БД.
2. **Каталог типов — остаётся в БД (расширить).** `document_type` уже редактируемый оператором справочник «что документ *может* быть». Достаточно добавить `a1` и `posting` (§11.1).
3. **Правила — выносим из кода в конфигурацию.** «Какие типы *обязательны* для данного водителя», «при каких условиях», «пороги истечения по типу» — это **регуляторный ruleset**, который и меняется/расширяется. Его не следует хардкодить в `if`-ветках и не обязательно строить под него тяжёлые админ-таблицы.

### 12.2. Почему именно конфигурация-ruleset, а не «всё в БД» и не хардкод

В репозитории **уже есть** проверенный паттерн ровно для такого: регуляторные параметры лежат git-версионируемым YAML и грузятся в **неизменяемый реестр при старте** — `data/country_rates/*.yaml` (ставки по странам) и `data/tax_rules/pl_<year>.yaml` (налоги/ZUS), при битом файле приложение **отказывается стартовать** (CLAUDE.md, §«Rates & audit»). Тот же подход подходит для требований к комплектности:

- **Расширяемость:** добавить тип/правило = правка YAML + PR → код-ревью и история в git = бесплатный аудит «кто и когда поменял норматив».
- **Условность (главная сложность):** правило несёт **предикат** по атрибутам водителя — `applies_if`: гражданство ≠ ЕС → `visa`/`residence`/`a1`; груз содержит ADR → `adr`; маршрут включает DE → `posting`(BAG) и т.д. В коде это превратилось бы в разрастающийся каскад `if`.
- **Согласованность с архитектурой:** тот же `RateRegistry`-подобный реестр, та же конвенция «норматив = данные, загружаемые при старте».
- **Развязка:** правила отделены от фактических данных документов — норматив можно поменять, не трогая записи.

**Чисто в БД (админ-таблицы `requirement_profile` / `requirement_rule`, редактируемые из UI)** оправдано, только если **нетехнические операторы** должны сами часто менять нормы без деплоя. Для небольшой компании, где нормы меняются несколько раз в год по регуляторным обновлениям, YAML+PR лучше: ревью, история, воспроизводимость. (Опционально — read-only экран «текущий ruleset» в UI.)

**Чистый хардкод** отвергается прямо по причине из вопроса — правила меняются и расширяются.

### 12.3. Предлагаемая реализация (эскиз, на согласование)

- `data/document_requirements/driver.yaml` — список правил: `{ type, required: true|conditional, applies_if: {...}, thresholds: [120,60], untracked: false }`.
- Загрузка в неизменяемый реестр в фабрике приложения (рядом с rate/tax-реестрами); битый YAML → отказ старта.
- **Пороги истечения тоже сюда.** Сейчас они захардкожены (`app/documents/constants.py`: `GENERIC_THRESHOLDS`, `INSURANCE_THRESHOLDS`, `UNTRACKED_TYPES`). Перенос в ruleset делает политику сроков данными, а `document_status()` (§6) — её потребителем.
- **Контроль комплектности — чистая функция:** `(водитель, его документы, активный ruleset) → {в порядке | скоро истекает | не хватает}`. Без БД-доступа, как `document_status()`, → регистрируется Jinja-глобалом и рисует сводку прямо в карточке водителя (§12.5).

### 12.4. Виды правил (по решениям пользователя)

Каждому типу в ruleset присваивается **режим обязательности**:

- **`required`** — должен быть всегда; отсутствие → попадает в блок «не хватает». Базовый набор §11 (license, code95, medical, psychological, tacho_card, passport, employment).
- **`conditional`** — обязателен **при выполнении предиката** `applies_if` по атрибутам водителя. Гражданство ≠ ЕС → `visa`/`residence`/`a1`; работа/командирование в ЕС → `a1`; маршрут включает страну X → `posting`(система IMI X). Источник атрибутов: гражданство есть в карточке (§3.1); «маршруты/страны» в модели водителя пока **нет** — нужно поле/признак (остаётся уточнить).
- **`present_only`** (track-if-present) — **никогда не считается «не хватает»**, отслеживается **только если документ есть**. По решению пользователя сюда относится **ADR**: возит ли водитель опасные грузы, определяем по **наличию актуального ADR-документа**, а не по отдельному признаку «hazmat». Если актуальный `adr` есть — показываем его статус (в «в порядке» / «скоро истекает»); если нет — ничего не требуем. Это снимает потребность в отдельном атрибуте груза.

### 12.5. Размещение сводки в карточке водителя (по решению пользователя)

Сводка комплектности занимает **место текущей секции контрактов** на карточке водителя (`app/templates/drivers/show.html`) и показывает **три блока**:

1. **В порядке** — обязательные/применимые документы, которые есть и не на грани истечения (статус `ok`).
2. **Скоро истекают** — есть, но в зоне порога (`soon` / `urgent` / `critical` / `expired`), отсортированы по близости `end_date`.
3. **Не хватает** — `required`/применимые `conditional`, для которых у водителя **нет** актуального документа.

> **Требует подтверждения.** Текущая секция контрактов (§7) — это не только документы-договоры (`employment`), но и отдельная сущность «Контракт» (`base_salary_pln`, `hours_norm`), используемая будущим payroll. Если сводка встаёт **вместо** неё, контрактам нужно новое место (вкладка / перенос ниже), иначе их не будет видно. Уточнить: контракты переносим или сводка делит секцию с ними?

### 12.6. Триггер пересчёта

- **Статусы сроков и сводка комплектности — без триггера:** обе считаются чистой функцией от `date.today()` при каждом рендере карточки (как `document_status()` сейчас). Отдельный пересчёт/планировщик не нужен — всегда актуально на момент открытия.
- **Архивация** (перенос истёкших версий в `Archive/`, `archived_at`) — **единственный существующий триггер: конвейер загрузки** (`_run_archive_pass`, `app/documents/persistence.py`). Планировщика (cron) в проекте **нет**, поэтому истёкший дубль уходит в `Archive/` только при следующей загрузке, затрагивающей водителя (открытый пункт §9 п.12). На сводку комплектности это **не влияет** — она и так читает только `active_documents` и считает статусы вживую.
