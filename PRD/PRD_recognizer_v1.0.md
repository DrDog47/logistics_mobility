# PRD — Подход к распознаванию документов (Recogniser approach)

## Mobility Payroll · версия 1.0 (черновик для правки · 2026-06-10)

*Логистическая компания (г. Белосток, Польша)*

> **Назначение документа.** Этот PRD описывает **единый подход к распознаванию документов** — общий для всех сущностей компании (водители, ТС, в будущем — организации). Раньше эта тема дублировалась в §8.7 каждого предметного PRD; теперь она вынесена сюда, а предметные PRD на неё **ссылаются**:
> - [`PRD_driver_v1.0.md`](PRD_driver_v1.0.md) §8.7 → этот документ;
> - [`PRD_vehicle_v1.0.md`](PRD_vehicle_v1.0.md) §8.7 → этот документ.
>
> Здесь зафиксирован **общий механизм** (порт, две стадии, адаптеры, конфигурация, контракт результата). **Логика привязки** распознанного документа к субъекту (создание/обновление водителя по паспорту, ТС — по свидетельству о регистрации, сопоставление по ключам, защита от дублей) остаётся **в предметных PRD** (driver §8.4–§8.5, vehicle §8.4–§8.5), потому что она для каждой сущности своя.

---

## 1. Цель и принцип

Идентификация содержимого документа (тип, даты, ФИО / реквизиты ТС, идентификаторы) выполняется **LLM-моделью**, но любое обращение к ней спрятано за **интерфейсом распознавания** (`DocumentRecognizer`). Это позволяет **заменить модель/провайдера в любой момент** — в том числе перейти на **локальную модель** — не трогая конвейер обработки `_Inbox/` и привязки документов.

Принципы:

- **Провайдер-агностично.** Конвейер зависит только от интерфейса и **фиксированной схемы результата**, а не от конкретной модели/SDK.
- **Структурированный выход.** Реализация обязана возвращать структуру по схеме (structured output / tool use), а не свободный текст.
- **Подменяемость через конфиг.** Выбор адаптера и модели — переменными окружения, без правок кода.
- **Раздельные конвейеры привязки.** Стадии распознавания и переключение моделей — **общие**; бизнес-логика привязки к водителю/ТС — **раздельная** (§8).

## 2. Двухстадийный конвейер

Распознавание идёт в **две независимо подменяемые стадии** (`app/documents/recognizer/composite.py:TwoStageRecognizer`), скрытые за единым контрактом `recognize() → RecognitionResult`:

1. **Стадия 1 — идентификация (identification / recognition).** Классификация **типа** документа (сопоставление с каталогом `document_type`). Дешёвая/быстрая (или локальная) модель. Возвращает `IdentificationResult`.
2. **Стадия 2 — извлечение (extraction / parsing).** Чтение **полей** под уже распознанный тип (даты, ФИО, идентификаторы, реквизиты ТС). Более сильная модель. Возвращает `RecognitionResult`.

Правила:

- **Стадия 2 пропускается**, если стадия 1 не смогла классифицировать документ (`recognized = False`) — тогда файл идёт на ручной разбор (остаётся в `_Inbox/`).
- Каждая стадия **выбирает провайдера/модель независимо** (§6) — например, идентификация на `claude-haiku-4-5`, извлечение на `claude-sonnet-4-6`, либо обе на `fake`/локальной модели.

## 3. Порт и контракт

Пакет `app/documents/recognizer/`. Ключевые абстракции (`base.py`):

| Абстракция | Роль |
|------------|------|
| `DocumentRecognizer` | Порт: `recognize(content, mime_type) → RecognitionResult`. Единая точка для конвейера. |
| `DocumentIdentifier` | Стадия 1: классификация типа → `IdentificationResult`. |
| `DocumentFieldExtractor` | Стадия 2: извлечение полей для известного типа → `RecognitionResult`. |
| `TwoStageRecognizer` | Композиция identifier + extractor за портом `DocumentRecognizer` (`composite.py`). |
| `RecognitionResult` | Структурированный результат (frozen dataclass). |
| `IdentificationResult` | Промежуточный результат стадии 1 (тип + уверенность). |
| `RecognizerError` | Ошибка распознавания (конфиг/провайдер/транспорт). |

- **Вход:** содержимое файла (байты) + MIME-тип (PDF / JPG / JPEG / PNG). Формат не влияет на контракт — анализируется содержимое.
- **Выход:** `RecognitionResult` (см. §4). Все поля опциональны, кроме `recognized`; провайдер, не разобравший файл, возвращает `recognized = False` с пояснением `note`.
- **Контракт:** результат — всегда фиксированная структура, не свободный текст, чтобы конвейер привязки (§8) не зависел от модели.

## 4. Схема результата (`RecognitionResult`)

Минимальный набор полей структурированного выхода. Общие поля плюс специфика по сущности (стадия 2 заполняет то, что применимо к распознанному типу).

**Общие:**

- `recognized` (bool) — удалось ли распознать; `note` — пояснение/флаг «не наш документ».
- `entity_type` — `driver` / `vehicle`.
- `document_type` — предполагаемый код типа для сопоставления с каталогом (driver §8.5.1 / vehicle §8.5.1).
- `start_date`, `end_date`, `document_id` — срок действия и номер документа (если есть).
- уверенность / флаг распознавания.

**Специфика `entity_type = driver`:**

- `identification_id` — стабильный ключ водителя (driver §8.4);
- `passport_number` — номер паспортной книжки;
- `first_name`, `last_name`;
- `birth_date`, `nationality` (ISO 3166-1 alpha-3) — из паспорта.

**Специфика `entity_type = vehicle`:**

- `vin` — стабильный ключ ТС (vehicle §8.4);
- `registration_plate` — госномер (вспомогательная подсказка);
- `brand`, `model`, `vehicle_type` — из свидетельства о регистрации.

> Конкретные **правила определения и сопоставления идентификаторов** (как нормализовать `identification_id` / `vin`, как разрешать неоднозначности) будут уточнены **по ходу запуска обработки документов** на практике. Здесь фиксируется лишь, какие поля возвращает схема; ключи сопоставления — в предметных PRD (driver §8.4–§8.5, vehicle §8.4–§8.5).

## 5. Адаптеры (реализации интерфейса)

| Адаптер | Статус | Реализация | Примечание |
|---------|--------|-----------|-----------|
| Офлайн `fake` | по умолчанию (dev/tests) | `FakeRecognizer` (`fake.py`) | Не смотрит в содержимое — парсит имя файла по конвенции ТЗ `<...>_<Type>_<dates>.<ext>`. Детерминирован, без сети и ключа. |
| Облачный LLM (`claude`) | v1.0 (продакшен) | `ClaudeIdentifier` + `ClaudeExtractor` (`claude.py`) | Claude API: image/PDF + structured output. Ленивый импорт `anthropic`. Каждая стадия — свой адаптер и своя модель. |
| Локальная модель | будущее | — | Тот же интерфейс, без изменений в конвейере. |
| Ручной ввод | fallback | — | Когда автоматика не справилась — оператор заполняет поля вручную; файл остаётся в `_Inbox/`. |

Выбор и сборка адаптеров — фабрика `factory.py` (`build_recognizer` / `init_recognizer` / `get_recognizer`), строится один раз при старте приложения, по аналогии с реестром ставок. Тесты обязаны быть офлайн — `TestingConfig` принудительно ставит `fake`.

## 6. Конфигурация (env)

| Переменная | Назначение | По умолчанию |
|------------|-----------|--------------|
| `DOCUMENT_RECOGNIZER` | Фолбэк-провайдер для обеих стадий, если их собственный провайдер не задан | `fake` |
| `DOCUMENT_IDENTIFIER` | Провайдер стадии 1 (`fake` / `claude`) | (фолбэк на `DOCUMENT_RECOGNIZER`) |
| `DOCUMENT_IDENTIFIER_MODEL` | Модель стадии 1 | `claude-haiku-4-5` |
| `DOCUMENT_EXTRACTOR` | Провайдер стадии 2 (`fake` / `claude`) | (фолбэк на `DOCUMENT_RECOGNIZER`) |
| `DOCUMENT_EXTRACTOR_MODEL` | Модель стадии 2 | `claude-sonnet-4-6` |
| `ANTHROPIC_API_KEY` | Ключ облачного провайдера | — (только из env) |

Единственная `DOCUMENT_RECOGNIZER=claude` подключает обе стадии; стадии можно развести по моделям/провайдерам независимо.

## 7. Требования к слою распознавания

- **Подменяемость:** выбор адаптера/модели — через конфигурацию (env), без изменения кода конвейера. По умолчанию в продакшене — облачный LLM-API.
- **Изоляция секретов:** ключи API провайдера — только в переменных окружения, не в коде и не в репозитории (как `SECRET_KEY`).
- **Отказоустойчивость:** ошибка/таймаут провайдера не теряет файлы — они остаются в `_Inbox/` и помечаются «требует внимания»; обработку можно повторить.
- **Воспроизводимость/аудит:** желательно сохранять, какой провайдер/модель и когда обработали документ (для разбора спорных распознаваний). Конкретику — на этап реализации.
- **Приватность:** при облачном провайдере персональные данные (паспорта) уходят во внешний сервис — учесть в политике обработки ПДн; локальная модель в будущем снимает этот вопрос.

> Конкретный облачный провайдер/SDK и схема промпта/инструмента — деталь реализации, не фиксируется в PRD. Важно лишь, что всё это **за интерфейсом** `DocumentRecognizer`.

## 8. Раздельные конвейеры привязки (driver / vehicle)

Слой распознавания (стадии 1–2, порт, адаптеры, конфиг) — **общий**. Конвейер **привязки** распознанного документа к субъекту реализуется **отдельно для каждой сущности**, так как правила у них разные:

| Аспект | Driver | Vehicle |
|--------|--------|---------|
| Документ-триггер создания | Паспорт (`passport`) | Свидетельство о регистрации (`tech_passport`) |
| Стабильный ключ сопоставления | `identification_id` | `vin` |
| Вспомогательная подсказка | `passport_number`, ФИО | `registration_plate` |
| Хранение сканов | таблица `driver_file` (1-ко-многим) | массив `file_links JSONB` в `document` |
| Логика create/update/no-op | driver §8.4 | vehicle §8.4 |
| Сопоставление остальных документов | driver §8.5 | vehicle §8.5 |

Общее для обоих: запись идёт в **единую таблицу `document`** (single-table inheritance, дискриминатор `entity_type`), документ без опознанного субъекта или нераспознанного типа **остаётся в `_Inbox/`**, ни одна запись не создаётся без подтверждения пользователем.

## 10. Вне рамок / уточняется позже

- **Правила идентификации (ключи сопоставления)** — конкретные алгоритмы по `identification_id` / `vin`, разрешение неоднозначностей — формулируются **после старта обработки документов** на практике.
- **Хранение метаданных распознавания** (провайдер/модель/время) для аудита — на этап реализации.
- **Локальная модель** как ещё один адаптер — будущее.
- **Распознавание парности front/back** и полностью автоматическая запись без подтверждения — вне v1.0.
