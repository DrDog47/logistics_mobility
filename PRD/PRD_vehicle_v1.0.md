# PRD — Модуль «Транспорт» (ТС)

## Mobility Payroll · версия 1.0 (черновик для правки · 2026-06-10)

*Логистическая компания (г. Белосток, Польша)*

> Документ описывает **продуктовые требования** к модулю учёта транспортных средств (тягачей и прицепов): какие данные хранятся, какие действия доступны, кому и по каким правилам. Техническая схема хранения — в [`TDD_data_schema_v2.0.md`](TDD_data_schema_v2.0.md) (§6 `vehicles`; документы — в **единой** таблице `document`, см. ниже). Общий контекст системы учёта документов — в [`TZ_sistema_ucheta_dokumentov_v1.4.md`](TZ_sistema_ucheta_dokumentov_v1.4.md). Симметричный модуль водителей — в [`PRD_driver_v1.0.md`](PRD_driver_v1.0.md); общие механизмы (drag-and-drop, `_Inbox/`, распознавание, версионность) описаны там и здесь переиспользуются с оговорками для ТС.

> **Важно об архитектуре документов.** Отдельной таблицы `vehicle_document` **больше нет**. Документы всех сущностей компании хранятся в **одной унифицированной таблице `document`** (single-table inheritance: базовый класс `Document`, дискриминатор `entity_type`, подклассы `DriverDocument` / `VehicleDocument` — `app/documents/models.py`, миграция `a7b8c9d0e1f2_unify_document_table`). Документ ТС — это строка в `document` с `entity_type = 'vehicle'`, заполненным `vehicle_uuid` (и `driver_uuid = NULL`). Сканы ТС хранятся массивом `file_links JSONB` в той же строке (у водителя сканы вынесены в отдельную таблицу `driver_file`; ТС пока сохраняет более простую форму).

## 1. Контекст и цель

Компания занимается грузоперевозками (автопарк ~24 фуры плюс прицепы). Каждое транспортное средство должно иметь действующий пакет документов с контролируемыми сроками — без действующей страховки OC или с просроченным техосмотром машина не может легально выезжать на маршрут. Модуль «Транспорт» — это единый реестр ТС: карточка с регистрационными данными, привязка к организации-владельцу и пакет документов с контролем сроков.

### 1.1. Целевой результат

- Единая карточка ТС со всеми регистрационными и техническими данными.
- Привязка ТС к организации (§3 PRD-схемы, FK `organisation_uuid`).
- Пакет документов ТС с автоматическим расчётом сроков и подсветкой истекающих (см. модуль документов); для всех отслеживаемых документов ТС — единый, более частый график алертов 60/30/15/0 (§11.2).
- Минимум ручного ввода при загрузке документов: пакетная загрузка перетаскиванием (drag-and-drop) — см. §8.

### 1.2. Отличия от модуля водителей

| Аспект | Водитель | Транспорт |
|--------|----------|-----------|
| Бизнес-ключ (стабильный идентификатор) | `identification_id` (личный номер) | **`vin`** (17 симв., уникален) |
| Документ-«паспорт» (триггер создания) | Паспорт (`passport`) | **Свидетельство о регистрации** (`tech_passport`, §8.4) |
| Отпуска / отсутствия | есть (отдельный модуль) | **нет** — у ТС секции отпусков не существует |
| Контракты | есть (§7 PRD водителя) | **нет** — у ТС контрактов нет |
| График алертов | страховка — 60/30/15; остальное — 120/60 | **единый для всех документов ТС** — 60 / 30 / 15 / 0 дней (§11.2) |
| Хранение сканов документа | таблица `driver_file` (1-ко-многим) | массив `file_links JSONB` в строке `document` (подкласс `VehicleDocument`) |

> И документы водителя, и документы ТС лежат в **одной** таблице `document`; различаются дискриминатором `entity_type` (`driver` / `vehicle`) и тем, какой FK заполнен (`driver_uuid` / `vehicle_uuid`).

## 3. Карточка ТС

Сущность **Транспортное средство** объединяет тягачи (`tractor`) и прицепы (`trailer`). Поля — по TDD §6.

### 3.1. Поля

| Поле | Обяз. | Правило / формат | Описание |
|------|:----:|------------------|----------|
| Тип ТС (`vehicle_type`) | да | `tractor` / `trailer` | Тягач (фура) или прицеп |
| VIN (`vin`) | да | 17 симв., **уникален** | Бизнес-ключ ТС (как правило в верхнем регистре) |
| Марка (`brand`) | да | до 100 симв. | Например, `Volvo`, `MAN` |
| Модель (`model`) | да | до 100 симв. | — |
| Регистрационный знак (`registration_plate`) | да | до 20 симв. | Госномер; в верхнем регистре |
| Страна регистрации (`registration_country`) | нет | ISO 3166-1 alpha-3 (3 симв.) | Страна, выдавшая регистрацию; **распознаётся из свидетельства о регистрации** (`tech_passport`, §8.4). Та же конвенция, что `nationality` водителя |
| Дата регистрации (`registration_date`) | нет | дата (формат `ГГГГ-ММ-ДД`) | Дата регистрации ТС; берётся из свидетельства о регистрации |
| Дата приобретения (`acquisition_date`) | нет | дата | — |
| Дата выпуска (`manufacture_date`) | нет | дата | — |
| Активно (`is_active`) | — | по умолч. `да` | ТС в эксплуатации |
| Организация (`organisation_uuid`) | да | выбор из списка организаций | Владелец |
| `extra` | нет | JSON | Резерв под расширяемые атрибуты |

> **Отображаемое имя** ТС формируется как `<Brand>_<Plate>` (латиницей), например `Volvo_WB1234A` — это же имя используется для папки в хранилище (§8.6). Отдельного поля нет (вычисляется).

> **Примечание.** В отличие от водителя (где `organisation_uuid` сделан nullable ради автосоздания из паспорта, PRD водителя §8.4), у ТС организация по TDD §6 — `NOT NULL`. Поведение при автосоздании из свидетельства о регистрации — см. §8.4 и открытый вопрос §9 п.6.

### 3.2. Связи

- **Организация** — каждое ТС относится к одной организации; организацию нельзя удалить, пока на неё ссылаются ТС (FK `RESTRICT`).
- **Документы** — ноль или несколько (`VehicleDocument` — подкласс `Document`, строки в общей таблице `document` с `entity_type = 'vehicle'`), сортируются по дате окончания.
- **Рейсы** — ноль или несколько (`trips.vehicle_id`, `SET NULL` при удалении ТС; вне рамок данного PRD).
- **Контрактов и отпусков у ТС нет** (§1.2).

### 3.3. Допустимые значения `vehicle_type`

| Значение | Описание |
|----------|----------|
| `tractor` | Тягач (фура, грузовик > 3,5 т) |
| `trailer` | Прицеп |

`vehicle_type` — `VARCHAR`, не `ENUM`, для симметрии с `document_type` (TDD §6.3).

## 4. Сценарии использования

### 4.1. Список ТС
Любой аутентифицированный сотрудник видит список, отсортированный по типу ТС, затем по госномеру.

### 4.2. Живой поиск
Поле поиска фильтрует по подстроке (без учёта регистра) по полям: госномер (`registration_plate`), VIN, марка, модель. Поиск работает «на лету» (HTMX-фрагмент таблицы, без перезагрузки страницы).

### 4.3. Создание ТС
Форма со всеми полями §3.1. После сохранения — переход на карточку ТС, сообщение «Vehicle … added».

> **Условие создания (требование пользователя).** Для создания нового ТС нужен документ — **свидетельство о регистрации** (`tech_passport`, аналог паспорта водителя). Ручное создание формой допускается, но «штатный» путь — распознавание свидетельства о регистрации из пакета документов (§8.4), откуда берутся VIN, марка, модель, госномер.

### 4.4. Просмотр карточки
Карточка показывает: регистрацию (тип, VIN, марка, модель, госномер), организацию-владельца, пакет документов (с бейджами сроков), сводку комплектности (§12) и зону drag-and-drop под секцией документов (§8.1).

### 4.5. Редактирование
Изменение полей карточки. Сообщение «Vehicle updated».

### 4.6. Удаление (мягкое)
Удаление помечает запись как удалённую (soft delete), физически не стирает. Удалённые ТС не показываются в списке/поиске и недоступны по прямой ссылке (404). **Все удаления в модуле — мягкие** (требование пользователя).

## 5. Бизнес-правила

1. **Уникальность.** `vin` — уникален в пределах системы. Повторная запись с тем же VIN не создаётся (это и есть стабильный ключ сопоставления, §8.4).
2. **Обязательный владелец.** ТС не может существовать без организации-владельца (FK `NOT NULL`, TDD §6).
3. **Мягкое удаление.** Записи не стираются; каждый список/поиск/карточка работают только с активными (`is_deleted = false`). Удаление обратимо на уровне данных.
4. **VIN — стабильный ключ, госномер — нет.** Госномер (`registration_plate`) может меняться (перерегистрация, смена владельца) и ключом сопоставления **не служит** — аналогично номеру паспортной книжки у водителя (§8.4).
5. **Даты в UTC** на уровне хранения; в интерфейсе — локальная дата.
6. **У ТС нет контрактов и отпусков** — секции из модуля водителей здесь не применяются (§1.2).

## 8. Планируемое в v1.0 — массовая загрузка документов ТС (drag-and-drop)

> Механизм идентичен модулю водителей (PRD водителя §8) и переиспользует те же компоненты: партиал drag-and-drop, endpoint `POST /documents/upload`, папку `_Inbox/`, порт `DocumentRecognizer`. Ниже — только специфика ТС; общие детали — в PRD водителя.

### 8.1. Что нужно

Добавить **зону drag-and-drop** на **карточке конкретного ТС** — **под секцией документов** (требование пользователя). Это ТС предлагается по умолчанию для всех файлов пакета.

- Поскольку поток документов ТС менее «смешанный», чем у водителей, для v1.0 достаточно зоны **на карточке ТС**. Общую зону на странице списка ТС (для пакета документов разных машин, как у водителей) можно добавить позже (§9 п.1).
- В зону можно перетащить **большой пакет документов сразу** (множественная загрузка: сканы свидетельств о регистрации, полисов OC, техосмотров, зелёных карт).

> **Файлы загружаются в `_Inbox/`** (как для водителей, §8.6): drag-and-drop сначала складывает файлы в `_Inbox/` без записи в БД и без распознавания, и только затем они обрабатываются.

### 8.2. Пользовательский сценарий

1. Пользователь открывает карточку ТС и перетаскивает в зону под документами пакет файлов.
2. Файлы **сначала загружаются в `_Inbox/`** (смонтированный том, §8.6) и только **затем обрабатываются** — приём пакета отделён от распознавания.
3. Для каждого файла система пытается **распознать**: тип документа, ключевые даты (срок действия / окончания) и **к какому ТС он относится** (по VIN / госномеру в распознанных данных). По умолчанию предлагается ТС, с карточки которого идёт загрузка.
4. Система показывает результат на одном **редактируемом экране подтверждения**: «файл → ТС → тип → даты».
5. Пользователь подтверждает или правит по каждому файлу: привязку к ТС, тип, даты. Нераспознанные/неоднозначные файлы помечаются и требуют ручного выбора.
6. После подтверждения для каждого файла создаётся строка `document` (`entity_type = 'vehicle'`) соответствующего ТС; сроки сразу попадают в контроль (бейджи по единому графику ТС **60/30/15/0** для всех отслеживаемых документов — OC, техосмотр, зелёная карта; §11.2).

### 8.3. Требования к функции

- **Множественная загрузка**: один жест перетаскивания принимает много файлов.
- **Привязка к ТС**: ТС с карточки предлагается по умолчанию; привязку можно переопределить для отдельных файлов (на случай чужих документов в пакете).
- **Сопоставление с существующими ТС**: распознанное ТС сопоставляется с реестром по бизнес-ключу — **VIN** (приоритет), затем госномер как вспомогательная подсказка (§8.5).
- **Подтверждение перед записью**: ни одна запись не создаётся без шага подтверждения.
- **Права**: загрузка доступна ролям admin / accountant / fleet_manager (§6).

### 8.4. Логика по распознанному свидетельству о регистрации: создать / обновить / пропустить

> **Аналог §8.4 PRD водителя.** Распознанное **свидетельство о регистрации** (`tech_passport`, «Dowód rejestracyjny») управляет не только записью документа, но и **самой карточкой ТС** — это документ-триггер создания ТС (требование пользователя «для создания нового ТС нужно свидетельство о регистрации»).

**Ключ сопоставления — VIN** (поле `vin`, уникальное). VIN не меняется в течение жизни ТС и распознаётся стабильно. Госномер (`registration_plate`) при перерегистрации/смене владельца меняется и **надёжным ключом не служит**.

По результату распознавания свидетельства о регистрации система выбирает одно из трёх действий:

| Условие | Действие |
|---------|----------|
| ТС с таким **VIN нет** в базе | **Создать новое ТС** из распознанных данных (VIN, марка, модель, госномер, тип ТС, дата выпуска, **страна регистрации `registration_country`**, **дата регистрации `registration_date`**) |
| ТС **есть**, и данные регистрации **изменились** (новый госномер и т.п.) | **Обновить** данные карточки ТС (госномер и пр.), VIN остаётся прежним |
| ТС **есть**, и данные регистрации **совпадают** | **Ничего не делать** (no-op) — повторная загрузка того же свидетельства не плодит дублей |

Требования к этой логике:

- Действие (создать / обновить / пропустить) показывается **на экране подтверждения** до записи; пользователь может его отменить или поправить привязку.
- При **создании нового ТС** заполняются только данные, распознанные из свидетельства (VIN, марка, модель, госномер, тип ТС, дата выпуска, страна регистрации `registration_country` — ISO 3166-1 alpha-3, дата регистрации `registration_date` — `ГГГГ-ММ-ДД`).
- **Организация (`organisation_uuid`)** при автосоздании, как правило, из свидетельства не определяется. По TDD §6 поле `NOT NULL` — поэтому либо его заполняют вручную перед сохранением на экране подтверждения, либо требуется сделать поле nullable (как у водителя). **Решение — открытый вопрос §9 п.6.**
- Распознавание типа ТС (`tractor` / `trailer`): свидетельство о регистрации прицепа отличается от свидетельства тягача (категория ТС в самом документе); тип подставляется в `vehicle_type`, при неоднозначности правится вручную.

### 8.5. Привязка остальных документов и определение типа

После обработки свидетельств о регистрации (§8.4) система проходит по **остальным** документам пакета (полисы OC, техосмотры, зелёные карты).

Для каждого не-регистрационного документа система:

- **извлекает идентифицирующие данные ТС**: VIN и/или госномер;
- **сопоставляет с ТС** в реестре по приоритету:
  1. по `vin` (если распознан) — основной и надёжный ключ;
  2. иначе по `registration_plate` (при неоднозначности файл помечается «требует ручного выбора»);
- **определяет тип документа** (страховка OC, техосмотр, зелёная карта) — сопоставление с каталогом `document_type` (`entity_type = "vehicle"`), §8.5.1;
- **определяет даты действия** документа (начало и окончание);
- если есть, извлекает номер документа (`document_id`);
- создаёт строку `document` (подкласс `VehicleDocument`) у найденного ТС — **после подтверждения** на общем экране.

#### 8.5.0. Модель «документ ↔ файлы» (отличие от водителей)

В отличие от водителя (где сканы вынесены в таблицу `driver_file`, 1-ко-многим), у ТС сканы документа хранятся **массивом `file_links JSONB`** прямо в строке `document` (на подклассе `VehicleDocument`). Несколько сторон/страниц одного документа — несколько путей в этом массиве. Отдельной таблицы `vehicle_file` в v1.0 нет.

- Несколько файлов одного пакета с одинаковыми «ТС + тип + сроки» группируются в одну строку `document` (файлы складываются в `file_links`).
- Файл, которому не удалось присвоить ТС или тип, **остаётся в `_Inbox/`** для ручного разбора (строка `document` не создаётся).

#### 8.5.1. Сопоставление типа документа с каталогом

Распознанный тип сопоставляется с активными записями каталога `document_type` для `entity_type = "vehicle"`:

- базовые коды ТС: `tech_passport` (свидетельство о регистрации, бессрочный), `inspection` (техосмотр), `insurance` (страховка OC);
- новый код `green_card` (зелёная карта) — §11.1;
- таблица `document` ссылается на `document_type` составным FK `(document_type, entity_type)` с `ondelete RESTRICT` (`fk_document_type` в `app/documents/models.py`) — документ **невозможно записать с типом, которого нет в каталоге**;
- если соответствие не найдено — документ **не записывается** и остаётся в `_Inbox/` для ручного разбора. Автосоздание новых типов в v1.0 не выполняется.

#### 8.5.2. Определение дат действия документа

- `start_date` — дата начала действия (выдачи / «valid from»);
- `end_date` — дата окончания действия (срок / «valid to»);
- оба поля опциональны в модели; **`end_date` — ключ контроля сроков**;
- свидетельство о регистрации (`tech_passport`) — **бессрочное**, не отслеживается (`UNTRACKED_TYPES`), даты могут отсутствовать;
- техосмотр (`inspection`) — обычно только дата окончания (следующего осмотра);
- страховка OC (`insurance`) и зелёная карта (`green_card`) — период, обе даты;
- если у отслеживаемого документа `end_date` не распознан — он выносится на ручную проверку (срок без даты контролировать нельзя);
- если распознаны обе даты и `end_date < start_date` — пара ошибочна, требует ручной правки.

#### 8.5.3. Запись в таблицу `document` (entity_type = vehicle)

После подтверждения для каждого документа создаётся строка в **единой таблице `document`** (через подкласс `VehicleDocument`), привязанная к `uuid` нужного ТС:

| Поле (колонка `document`) | Значение |
|------|----------|
| `uuid` | Генерируется автоматически (PK) |
| `entity_type` | `"vehicle"` — дискриминатор single-table inheritance; вместе с `document_type` образует составной FK в каталог |
| `vehicle_uuid` | **UUID ТС** (FK → `vehicles.uuid`, `RESTRICT`); `driver_uuid` остаётся `NULL` |
| `document_type` | Код типа из каталога (§8.5.1) |
| `document_id` | Номер самого документа, если распознан (например, номер полиса OC) |
| `start_date` / `end_date` | Даты действия (§8.5.2) |
| `file_links` | Ссылка(и) на скан(ы) — JSON-массив (колонка подкласса `VehicleDocument`) |
| `archived_at` | Заполняется при переносе устаревшей версии в `Archive/` (§8.6) |
| `extra` | Тип-специфичные атрибуты: для `insurance` — `insurance_company`; для `inspection` — `periodicity_months` (по умолч. 12) |

Правила записи — как у водителей: транзакционность пакета, целостность ссылок (`vehicle_uuid` указывает на активное ТС, `document_type` существует в каталоге), контроль сроков включается сразу после записи.

**Защита от дублей.** Перед записью система ищет у ТС существующий документ того же типа с теми же сроками (`document_type` + `start_date` + `end_date` при `entity_type = 'vehicle'` и совпадающем `vehicle_uuid`):
- дубль найден, данные ТС не изменились → no-op;
- дубль найден, данные ТС изменились → обновить;
- дубля нет → создать новую строку.
Добавление второй страницы/стороны того же документа — это пополнение `file_links`, не дубль.

### 8.6. Хранение файлов (локально, docker volume)

Идентично PRD водителя §8.6: файлы хранятся локально, внешняя папка через docker volume mount, двухфазная загрузка через `_Inbox/`. Документы ТС раскладываются в ветку `Transport/` иерархии:

```
<DOCUMENTS_DIR>/
├── _Inbox/                          ← точка входа для новых сканов (общая)
├── Drivers/
│   └── <Name_Surname>/ …
└── Transport/
    └── <Brand_Plate>/               ← напр. Volvo_WB1234A (латиницей)
        ├── <Brand_Plate>_<Type>_<dates>.<ext>
        └── Archive/                 ← устаревшие версии (не удаляются)
```

- **Правило именования** (из ТЗ §3): `<Владелец>_<Тип>_<дата(ы)>.<ext>`; даты `ГГГГ-ММ-ДД`; документы со сроком получают две даты `начало_конец`. Латиницей. Примеры: `Volvo_WB1234A_Insurance_2025-01-01_2026-01-01.pdf`, `Volvo_WB1234A_Inspection_2026-08-15.pdf`, `Volvo_WB1234A_TechPassport.pdf`.
- **Связь с БД**: в `document.file_links` хранятся **пути внутри иерархии** (относительно `DOCUMENTS_DIR`), а не blob'ы.
- **Версионность**: два документа на одно и то же (например, два действующих полиса OC) сосуществуют до истечения раньше истекающего; по факту истечения он уходит в `Archive/` (проставляется `archived_at`), второй остаётся; основной в реестре — с самой поздней `end_date`. Уже просроченный при загрузке (при наличии более свежего) кладётся сразу в `Archive/`.

### 8.7. Распознавание документов за интерфейсом

> **Вынесено в отдельный PRD.** Подход к распознаванию (провайдер-агностичный порт `DocumentRecognizer`, две стадии «идентификация → извлечение», адаптеры `fake`/`claude`/локальная/ручной ввод, конфигурация стадий, контракт результата) — **общий для водителей и ТС** и описан в [`PRD_recognizer_v1.0.md`](PRD_recognizer_v1.0.md). Конвейер распознавания документов ТС реализуется **отдельной веткой** (`entity_type = vehicle`), но теми же двумя стадиями и за тем же портом — см. [`PRD_recognizer_v1.0.md`](PRD_recognizer_v1.0.md) §2, §8.

Здесь, в PRD ТС, остаётся только **специфика привязки** распознанного к ТС:

- **Документ-триггер создания** ТС — свидетельство о регистрации (`tech_passport`); логика create / update / no-op — §8.4.
- **Стабильный ключ сопоставления** — `vin` (§8.4); госномер (`registration_plate`) — вспомогательная подсказка.
- **Сопоставление остальных документов** (OC, техосмотр, зелёная карта) и определение типа — §8.5; запись — в `document` через `VehicleDocument` (§8.5.3).
- Поля результата для ТС (`entity_type = vehicle`, `vin`, `registration_plate`, `brand`, `model`, `vehicle_type`, `registration_country` — ISO 3166-1 alpha-3, `registration_date`, `document_type`, даты, `document_id`) — см. схему в [`PRD_recognizer_v1.0.md`](PRD_recognizer_v1.0.md) §4.

> **Правила идентификации (ключи сопоставления) — формулируются позже.** Конкретные алгоритмы по `vin` / госномеру и разрешение неоднозначностей уточняются **после запуска обработки документов** на практике (общий принцип — в [`PRD_recognizer_v1.0.md`](PRD_recognizer_v1.0.md) §4, §10).

## 10. Нефункциональные требования

- **Интернационализация**: интерфейс на PL / EN / RU (Flask-Babel), по умолчанию PL.

## 11. Минимальный пакет документов ТС (нормативный список)

> **Требование пользователя.** Для легального выезда тягача/прицепа на маршрут нужен фиксированный минимум документов. Платформа должна (1) контролировать **наличие** набора и (2) отслеживать **сроки истечения** (§12).

| # | Документ | Код типа (`document_type`) | Срок / периодичность | Обязателен (условие) | Контроль срока |
|--:|----------|----------------------------|----------------------|----------------------|:--------------:|
| 1 | Свидетельство о регистрации (Dowód rejestracyjny / Vehicle registration certificate) — признаётся во всех странах ЕС по директиве 1999/37/EC | `tech_passport` | бессрочно (документ-карточка) | всем ТС | **нет** (бессрочный) |
| 2 | Полис OC (obowiązkowe ubezpieczenie OC posiadaczy pojazdów) — обязательное страхование гражданской ответственности | `insurance` | обычно 1 год | всем ТС | **да — 60/30/15/0** (§11.2) |
| 3 | Сертификат техосмотра (Przegląd techniczny) — периодический техосмотр по директиве 2014/45/EU; для коммерческих ТС > 3,5 т — ежегодно | `inspection` | 12 мес. (по умолч.) | всем ТС | **да — 60/30/15/0** (§11.2) |
| 4 | Зелёная карта (Green Card / Zielona Karta) — для международных перевозок, особенно за пределы ЕС | **`green_card`** (новый) | по сроку карты | при международных рейсах (вне ЕС) | **да — 60/30/15/0** (§11.2); режим `present_only` (§9 п.12) |

> Свидетельство о регистрации — **аналог паспорта водителя**: документ-триггер создания ТС (§8.4) и его «личность». Полис OC, техосмотр и зелёная карта — **отслеживаемые** документы; **все они используют единый, более частый график алертов 60/30/15/0** (§11.2), чтобы оставить окно на действия (переговоры/смену страховой, запись на техосмотр). Свидетельство о регистрации не отслеживается (бессрочное).

### 11.1. Новые типы документов в каталоге

В каталог `document_type` (`entity_type = "vehicle"`) к существующим (`tech_passport`, `inspection`, `insurance` — TDD §7.3, `app/documents/constants.py:VEHICLE_DOCUMENT_TYPES`) добавляется:

- **`green_card`** — «Green Card (Zielona Karta)». Документ-период (`start_date`/`end_date`), отслеживается; режим обязательности `present_only` (§12.4). Допускается в `extra` указывать перечень покрываемых стран.

> Свидетельство о регистрации использует существующий код **`tech_passport`** (метка «Tech passport» / «Свидетельство о регистрации») — отдельный новый тип не нужен.

### 11.2. Единый график алертов для всех документов ТС

> **Требование пользователя.** Для **всех** отслеживаемых документов ТС действует **один** график алертов — тот же, что у OC: **60 / 30 / 15 / 0 дней**. Длинные окна (60, 30) дают время на действия (переговоры/смену страховой, запись на техосмотр и т.п.).

| Порог | Уровень | Смысл |
|------:|---------|-------|
| 60 дней | `soon` (жёлтый) | «Скоро» — пора планировать продление / сравнивать предложения |
| 30 дней | `urgent` (красный) | «Срочно» — действовать |
| 15 дней | `critical` | «Критично» — последнее окно |
| 0 дней (`end_date < today`) | `expired` | «Просрочено» — ТС не может легально выезжать |

Применяется ко всем отслеживаемым типам ТС: `insurance` (OC), `inspection` (техосмотр), `green_card` (зелёная карта). Свидетельство о регистрации (`tech_passport`) не отслеживается (`UNTRACKED_TYPES`).

> **Замечание по реализации.** Сейчас график 60/30/15 в `document_status()` (`INSURANCE_THRESHOLDS`) привязан к **коду типа** `insurance`, а остальные типы идут по общей шкале 120/60. Чтобы применить 60/30/15/0 ко **всем** документам ТС, нужно изменить выбор шкалы: либо по **`entity_type == "vehicle"`** (все документы ТС → 60/30/15), либо — предпочтительно — брать пороги из ruleset `vehicle.yaml` (§12.1), где у каждого типа ТС `thresholds: [60, 30, 15]`. Это расхождение с PRD водителя (там у страховки была своя тесная шкала, у остального — 120/60); в модуле ТС единая шкала для всех.

## 12. Контроль комплектности и истечения сроков (подход)

Подход тот же, что для водителей (PRD водителя §12): три слоя — **факты** (строки `document` с `entity_type = 'vehicle'` в БД), **каталог типов** (`document_type` в БД, расширить кодом `green_card`), **правила** (версионируемый YAML-ruleset, грузится в неизменяемый реестр при старте; битый файл → отказ старта).

### 12.1. Ruleset для ТС

- `data/document_requirements/vehicle.yaml` — список правил по аналогии с `driver.yaml`: `{ type, mode, label, thresholds }`.
- Режимы обязательности (§12.4 PRD водителя):
  - **`required`** — должен быть всегда; отсутствие → «не хватает». Для ТС: `tech_passport` (наличие, без срока), `insurance`, `inspection`.
  - **`present_only`** — отслеживается, только если документ есть; никогда не «не хватает». Для ТС: `green_card` (зависит от маршрута, а признака маршрута в модели ТС пока нет — §9 п.12).
- Пороги (`thresholds`): **единый график 60/30/15/0 для всех отслеживаемых типов ТС** (§11.2) — у `insurance`, `inspection` и `green_card` одинаково `[60, 30, 15]`; `tech_passport` — без контроля срока (`untracked`).

Эскиз `vehicle.yaml`:

```yaml
version: 1
rules:
  - type: tech_passport
    mode: required
    label: Vehicle registration certificate
    untracked: true           # бессрочный — контролируется только наличие
  - type: insurance
    mode: required
    label: OC insurance
    thresholds: [60, 30, 15]   # единый график ТС (§11.2)
  - type: inspection
    mode: required
    label: Technical inspection
    thresholds: [60, 30, 15]   # единый график ТС (§11.2)
  - type: green_card
    mode: present_only
    label: Green Card
    thresholds: [60, 30, 15]   # единый график ТС (§11.2)
```

### 12.2. Сводка комплектности на карточке ТС

Сводка — чистая функция `(ТС, его документы, активный ruleset) → {в порядке | скоро истекает | не хватает}` — без БД-доступа, регистрируется Jinja-глобалом (как `document_status()`), рисует три блока на карточке ТС (вместо отсутствующей секции контрактов, §7):

1. **В порядке** — обязательные/применимые документы, которые есть и не на грани истечения.
2. **Скоро истекают** — есть, но в зоне порога (отсортированы по близости `end_date`); все документы ТС — по единому графику 60/30/15/0 (§11.2).
3. **Не хватает** — `required` типы, которых у ТС нет действующих (`green_card` в режиме `present_only` сюда не попадает).

### 12.3. Триггер пересчёта

- **Статусы сроков и сводка комплектности — без триггера:** считаются чистой функцией от `date.today()` при каждом рендере карточки. Всегда актуально на момент открытия.
- **Архивация** (перенос истёкших версий в `Archive/`, `archived_at`) — триггерится конвейером загрузки; планировщика (cron) в проекте нет (§9 п.11). На сводку не влияет — она читает только действующие документы.
